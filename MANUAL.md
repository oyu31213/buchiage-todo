# MANUAL.md — Claude Code でアプリを作る手順書

この手順書は、**ソフトウェアエンジニアでない人**が Claude Code を使って
Todo アプリを作るための進め方をまとめたものです。上から順に進めてください。

## この進め方の考え方（最初に読んでください）

あなた（人間）は **コードを書きません。読めなくても大丈夫です。** 代わりに、次の2つだけを担当します。

1. **何を作るか／作らないか**（スコープ）を決める
2. **どうなれば完成か**（受け入れ条件）を決め、最後に自分で触って確認する

実装やテストの中身は Claude に任せます。あなたは「やりたいこと」と「できているかどうか」を
**画面を見て・触って**判断します。各工程に「ここであなたが何を判断するか」を書いてあります。

---

## 0. 前提準備（最初に1回だけ）

- **Docker** が動く状態にする（PostgreSQL をローカルで動かすため）。
- **pnpm** を使えるようにする（パッケージ管理ツール）。
- **Pencil（MCP）** が Claude Code につながっているか確認する（デザイン作成に使う）。
- **skills.sh のスキルをインストールする。** ターミナルで、または Claude に頼んで実行：
  ```bash
  npx skills add clerk/skills@clerk-setup
  npx skills add clerk/skills@clerk-nextjs-patterns
  npx skills add vercel-labs/vercel-plugin@nextjs
  npx skills add vercel-labs/vercel-plugin@shadcn
  npx skills add bobmatnyc/claude-mpm-skills@drizzle-orm
  npx skills add pproenca/dot-skills@zod
  npx skills add dgalarza/claude-code-workflows@conventional-commits
  ```
- `.claude/settings.json` に、危険なコマンドを禁止する設定が入っていることを確認する（変更不要）。

> 💡 ターミナルで自分でコマンドを打ちたいときは、プロンプトの先頭に `!` を付けて
> `! npx skills add ...` のように打つと、この会話の中で実行できます。

---

## 1. 要件定義 — 何を作るか決める

Claude に「Todo アプリの要件を整理して」と頼みます（`requirements` スキルが動きます）。

- スコープ（やること）と**スコープ外（やらないこと）**がはっきりするまで、Claude の質問に答えます。
- **受け入れ条件**（例:「追加ボタンを押すと、入力したタスクが一覧に表示される」）が
  チェックリストになって `docs/requirements/` に保存されます。

👤 **あなたの判断**: やりたいこと・やらないことが合っているか。受け入れ条件に抜けがないか。

---

## 2. デザイン作成 — 見た目を決める

Claude に「デザインを作って」と頼みます（`design` スキルが Pencil を使います）。

- 画面のデザイン案ができます。「もっとこうしたい」を伝えて何度でも直してもらいます。

👤 **あなたの判断**: 見た目がイメージ通りか（コードではなく絵で判断できます）。

---

## 3. Plan mode で計画をレビュー — 作る前に確認する

実装に入る前に、Claude に **Plan mode**（Shift+Tab で切替）で「作業計画」を出してもらいます。

- 「何を・どの順番で作るか」が日本語で提示されます。
- 内容に納得したら承認します。違ったらこの場で修正を伝えます（**コードが書かれる前**に直せます）。

👤 **あなたの判断**: 計画が要件・デザインと合っているか。ここが1回目の重要なレビューポイントです。

---

## 4. 初回 scaffold — アプリの土台を作る（最初の1回だけ）

Claude に「Next.js でプロジェクトの土台を作って」と頼みます。
技術スタック（Next.js / Clerk / Drizzle / Zod / shadcn/ui）のスキルが働いて、
[CLAUDE.md](./CLAUDE.md) の構成規約どおりにフォルダと設定が用意されます。

👤 **あなたの判断**: 特になし（Claude に任せてOK）。

---

## 5. 実装 — 機能を作る

Claude に作りたい機能を頼みます。**一度に全部ではなく、小さく区切って**進めるのがコツです
（例:「まずタスクを追加・表示できるようにして」→次に「完了チェック」→次に「削除」）。

- 見た目は、shadcn/ui の部品を土台に **2のデザインに合わせて**作られます。

👤 **あなたの判断**: 特になし（この後の工程で確認します）。

---

## 6. テスト作成 — 自動で品質を守る仕組みを作る

Claude に「テストを書いて」と頼みます（`write-tests` スキル）。

- 受け入れ条件ごとに、ブラウザを自動操作して確かめる **E2E テスト**が作られます。
- 裏側のロジックには **unit テスト**が作られます。

👤 **あなたの判断**: 特になし（テスト結果は次の工程で見ます）。

---

## 7. コードレビュー — 品質を点検する

Claude Code の built-in コマンド **`/code-review`** を実行します（プロンプトに `/code-review` と入力）。
コードの品質を点検し、必要なら直します。より厳しく見たいときは `/review` や `/security-review` も使えます。

👤 **あなたの判断**: 特になし（点検は Claude が代行します）。

---

## 8. 手動確認 — 最後は自分で触る（いちばん大事）

Claude に「アプリを起動して」と頼み（`dev-server` スキル）、表示された URL（通常 http://localhost:3000）を
ブラウザで開きます。**あなた自身がアプリを触り**、受け入れ条件のチェックリストと、
2で作ったデザインを照らし合わせます。テスト（工程6）が通っていても、最後は必ず自分の目で確認します。

👤 **あなたの判断**: 自分の目と手で「できている」と言えるか。これが最終判断です。

---

## 9. コミット — 変更を記録する

Claude に「コミットして」と頼みます（Conventional Commits スキル）。
変更が `feat:` `fix:` などの形式で記録されます。

---

## うまくいかないとき

- Claude が同じ失敗を繰り返す → 一度立ち止まり、「今の状況」と「期待する結果」を具体的に伝え直す。
- エラーが出た → 画面に出た文言をそのまま Claude に伝える（例:「保存ボタンを押したら赤い文字で◯◯と出た」）。
- 危険なコマンドはそもそも実行できないように設定済みなので、安心して試してOKです。

> ⚠️ このプロジェクトでは「アプリを論理的に壊してしまったときの巻き戻し（git の使い方）」は扱いません。
> まずはこの手順を1周して、Claude Code で作る感覚をつかむことを目的にしています。
