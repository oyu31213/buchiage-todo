# TEACHING.md — 教え方マニュアル（配布・指導する人向け）

このファイルは、**このテンプレートを配布して人に教える「あなた（講師役）」**のためのガイドです。
学習者本人が読む手順書は [MANUAL.md](./MANUAL.md)。**このファイルは学習者には配らなくてOK**です
（配ってもいいですが、想定読者はあくまで教える側）。

> 学習者が実際にやる流れ → MANUAL.md
> プロジェクトの規約・技術スタック → CLAUDE.md
> あなたが「どう教えるか」 → このファイル

---

## 0. 誰に教えるための教材か

想定する学習者は **「フロントエンドは書けるが、バックエンド／インフラに不慣れな人」** です。

- できること: React / Next.js / TypeScript / UI 実装、コンポーネント設計。
- 不慣れなこと: **Docker、PostgreSQL、DB マイグレーション（Drizzle）、Server Actions、認証のサーバ側（Clerk）、`.env`、ポートやプロセスの話**。

この教材のねらいは「バックエンドを一から覚えさせること」**ではありません**。
**Claude Code に CLAUDE.md・スキル・Plan mode・MCP を与えると、自分が手薄な領域（バックエンド/インフラ）まで
含めて“動くアプリ”を作れる** — その体験をさせるのが目的です（CLAUDE.md「このプロジェクトについて」参照）。

👉 だから教えるときの主役は「コードの書き方」ではなく、**「Claude にどう任せ、どこを人間が判断するか」**です。

---

## 1. この教材で体験させたい4つの“効き目”

学習者に「これがあると無いとで出力が変わる」と体感させたいのは次の4つ。各工程でここに触れさせます。

| 仕掛け | 何が変わるか | どの工程で見せるか |
|---|---|---|
| **CLAUDE.md** | ファイルの置き場所・技術選定・規約が固定され、毎回ブレない | scaffold（工程4）以降ずっと |
| **スキル** | 「要件定義」「デザイン」「テスト」など作業が定型化され、頼むだけで動く | 各工程の頭 |
| **Plan mode** | 実装の**前**に計画を日本語でレビューでき、コードが書かれる前に直せる | 工程3（最重要レビュー点） |
| **MCP（Pencil）** | 画面デザインを“絵”で決められる（コードを読めなくても判断できる） | 工程2 |

> フロントの人は「コードは読める」ぶん、つい中身を読みにいって時間を溶かしがちです。
> **「読まずに、計画とデザインと受け入れ条件で判断する」**やり方の快適さを体験させてください。

## 2. 配布のしかた

このテンプレ一式（CLAUDE.md / MANUAL.md / TEACHING.md / docker-compose.yml / `.claude/` / docs/ など）を、
**学習者自身の GitHub リポジトリとして持たせる**のが基本です。

### 方法A: `start-new-repo` スキルで作らせる（推奨）

学習者に **「テンプレから自分のリポジトリを作って」**（または「start-new-repo」）と打たせます。
`start-new-repo` スキルが、`gh` でテンプレ `spyuchan/ai-coding-example` を clone → テンプレとの
つながりを外す → **本人のアカウントに新リポジトリを作成して push** まで一気にやります。

- 学習者に聞かれるのは **リポジトリ名**と **public/private** の2つだけ。
- 事前に **GitHub CLI のログイン**（`gh auth login`）が要る。ここは講師が一緒に済ませておくと詰まらない。
- 所有者は「`gh` でログイン中のアカウント」。**学習者自身のアカウントでログインしているか**を必ず確認する
  （あなたのアカウントのままだと、あなたの GitHub にできてしまう）。

### 方法B: clone / ZIP で渡す（gh を使わせない場合）

```bash
git clone <このリポジトリのURL> <学習者のプロジェクト名>
```

git を使わせたくない相手には ZIP で渡して展開させ、本人に `git init` から始めさせる
（コミット工程＝工程9の練習にもなる）。GitHub に上げる工程は別途案内する。

> どの方法でも、`.env.local` は**配らない／push しない**（git 管理外・実キーを含む想定）。
> 学習者に `.env.example` を `.env.local` にコピーさせ、キーは本人に入れさせること。

### 配る前に「前提準備」を必ず案内する

学習者の環境で次がそろっている必要があります（MANUAL.md「0. 前提準備」と同じ）。

- Docker（**起動まで**）、pnpm、Node
- skills.sh のスキル7つ（`clerk-setup` / `clerk-nextjs-patterns` / `nextjs` / `shadcn` / `drizzle-orm` / `zod` / `conventional-commits`）
- Pencil（MCP）の接続

> 💡 そろっているかは **`check-setup` スキル**で一発確認できます。学習者に
> **「環境が整っているか調べて」**と打たせれば、整っている／いないが表で出ます。最初の関門はここなので、
> 講師が画面を一緒に見て、足りないものを潰してから先に進むとスムーズです。

## 3. 教える流れ（MANUAL.md の各工程＋講師メモ）

MANUAL.md の「まず: 自分のリポジトリを用意する」＋工程 0〜9 に沿って進めます。
各工程で**「実演する／本人にやらせる／講師が見張る」**を分けて運用してください。

### まず: 自分のリポジトリを用意する（`start-new-repo`）
- 本人に「テンプレから自分のリポジトリを作って」と打たせる。リポジトリ名と public/private を本人が決める。
- 講師が見張る: **`gh auth login` 済みか**、そして **学習者本人のアカウントでログインしているか**
  （講師のアカウントのままだと講師の GitHub にできてしまう）。clone 先がテンプレの中に入れ子にならないかも見る。

### 工程0: 前提準備
- 本人に **`check-setup`**（「環境が整っているか調べて」）を打たせる。
- 講師が見張る: **Docker が起動しているか**、skills.sh の7スキル、Pencil 接続。ここが一番つまずく（→ 第5章）。

### 工程1: 要件定義（`requirements`）
- 本人に「Todo アプリの要件を整理して」と頼ませる。Claude の質問に**本人が答える**。
- 講師メモ: フロントの人は実装に飛びたがる。**「作らないこと（スコープ外）」を口に出させる**のがコツ。
  受け入れ条件が `docs/requirements/` にチェックリストで残ることを見せる。

### 工程2: デザイン（`design` / Pencil）
- 「デザインを作って」。Pencil で画面案が出る → **絵を見て**直しを伝えさせる。
- 講師メモ: ここで「コードを読まずに判断する」感覚を掴ませる。Pencil 未接続だとここで詰まるので工程0で確認済みに。

### 工程3: Plan mode で計画レビュー ← **最重要**
- 実装前に Plan mode（Shift+Tab）で「作業計画」を日本語で出させ、**本人が承認/修正**。
- 講師メモ: フロントの人でも**バックエンド側の計画（DB スキーマ・マイグレーション・Server Actions）は判断しづらい**。
  ここで「分からない部分は Claude に根拠を説明させる」「要件・デザインと食い違ってないかだけ見る」と教える。
  **コードが書かれる前に直せる**のがこの工程の価値。

### 工程4: scaffold（土台作り）
- 「Next.js でプロジェクトの土台を作って」。CLAUDE.md の構成規約どおりにフォルダ・設定ができる。
- 講師メモ: 生成物を一緒に眺め、「**置き場所が CLAUDE.md で決まっている**」ことを実感させる。
  ここで **`.env.local` の作成**（`.env.example` をコピー）と **DB ポート**の話が出る（→ 第5章のポート競合）。

### 工程5: 実装
- 小さく区切って頼ませる（追加→完了チェック→削除…）。UI は shadcn/ui ＋工程2のデザイン。
- 講師メモ: フロントの人の得意分野。**Server Actions 経由で DB に触る**（component から直接触らない）規約だけ意識させる。

### 工程6: テスト（`write-tests`）
- 「テストを書いて」。受け入れ条件→E2E（Playwright）、裏側ロジック→unit（Vitest）。
- 講師メモ: 各テストがどの受け入れ条件に対応するかコメントで残る。**工程1の受け入れ条件に戻って対応づけ**させる。

### 工程7: コードレビュー（built-in `/code-review`）
- `/code-review` を実行。必要なら `/review`・`/security-review`。
- 講師メモ: 自前レビュースキルは無い。点検は Claude に任せ、出た指摘の意味を一緒に読む。

### 工程8: 手動確認 ← **いちばん大事**
- 「アプリを起動して」（`dev-server`）→ ブラウザで**本人が触る**。受け入れ条件＋工程2デザインと照合。
- 講師メモ: テストが通っても最後は自分の目。ここが「できた」を本人が宣言する瞬間。

### 工程9: コミット（Conventional Commits）
- 「コミットして」。`feat:` `fix:` などで記録される。

## 4. 教えるときのスタンス（コツ）

1. **「コードを読ませない」を徹底する。** フロントの人ほど読みにいく。判断材料は
   **計画（工程3）・デザイン（工程2）・受け入れ条件（工程1）**の3つ。読むのは時間の溶けどころ。
2. **苦手領域（バックエンド/インフラ）は Claude に説明させる。** 「なぜこのスキーマ？」「マイグレーションって何？」は
   その場で Claude に日本語で説明させると、教材として一石二鳥。
3. **詰まったら“状況＋期待結果”を具体的に言い直させる。** エラーは画面の文言をそのまま貼らせる（MANUAL.md「うまくいかないとき」）。
4. **壊す心配をさせない。** 危険コマンドは `.claude/settings.json` で deny 済み。安心して試させる。
   （ただし git の巻き戻しはこの教材の範囲外。まず1周“作る感覚”を掴ませるのが目的。）

## 5. よくある詰まりどころと対処（講師用トラブルシュート）

フロントの人が特にハマりやすい順。**多くは工程0で `check-setup` を通せば未然に防げます。**

| 症状 | 原因 | 対処 |
|---|---|---|
| `start-new-repo` でリポジトリが作れない | `gh` 未ログイン | `! gh auth login` を案内。`gh auth status` で確認 |
| リポジトリが**講師のアカウント**にできてしまった | `gh` が講師アカウントでログイン中 | 学習者本人のアカウントでログインし直す。`gh auth status` で誰でログイン中か確認 |
| DB が起動しない／`docker compose` がエラー | **Docker が起動していない** | Docker Desktop を起動。`check-setup` で「起動中」を確認 |
| スキルが期待通り動かない | skills.sh の7スキル未導入 | MANUAL.md の `npx skills add ...` を実行。`check-setup` で確認 |
| デザイン工程で Pencil が動かない | **Pencil(MCP) 未接続** | VS Code の Pencil 拡張を起動 → `claude mcp list` で `✔ Connected` を確認 |
| `address already in use` で DB が立たない | **5432 ポート競合**（既存 Postgres 等） | `dev-server` スキルが空きポートを自動探索し `DB_PORT` を設定する。5432 以外になったら、その番号を `.env.local` の `DATABASE_URL`（=`DB_PORT`）に反映させる（CLAUDE.md「ローカル DB のポート」参照） |
| アプリが DB に繋がらない | `.env.local` 未作成 or ポート不一致 | `.env.example` を `.env.local` にコピー。`DB_PORT` と起動した DB のポートを一致させる |
| `pnpm` が無いと言われる | pnpm 未インストール | pnpm を入れる（`npm`/`yarn` は使わない方針） |

> **ポート競合は「フロントの人が原因を想像しにくい」典型**です。「すでに別の Postgres が 5432 を使っている」と
> いう状況自体が馴染みがないので、`dev-server` が自動で別ポートに逃がす仕組みを**一度実演**しておくと安心します。

## 6. セッション前チェックリスト（講師）

配布・指導の前に、あなた自身の手元と学習者の環境で確認：

- [ ] 学習者の環境に Docker / pnpm / Node が入り、**Docker が起動**している
- [ ] skills.sh の7スキルが導入済み（`check-setup` で確認）
- [ ] Pencil(MCP) が `✔ Connected`
- [ ] 学習者が **自分のアカウントで** `gh auth login` 済み（`start-new-repo` を使う場合）
- [ ] 配布方法を決めてある（`start-new-repo` で各自に作らせる／clone・ZIP で渡す）
- [ ] `.env.example` → `.env.local` の作り方を見せる準備（実キーは学習者自身に入れさせる）
- [ ] 「コードを読ませない／判断は計画・デザイン・受け入れ条件で」を最初に宣言する
- [ ] つまずき所（第5章）を事前に一度自分で再現しておく（特にポート競合の実演）
