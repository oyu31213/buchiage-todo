# デザイン作成スキル

要件(`docs/requirements/`)が固まった後、**実装に入る前**に画面の見た目を決めます。
デザインは Pencil(MCP, `mcp__pencil`)で作成します。出来上がったデザインは
**「見た目の受け入れ条件」** として、後の実装・最終確認の基準になります。

## 前提

- このスキルは Plan mode のレビューの **前** に使う。要件とデザインが揃った状態で、
  人間が作業計画をまとめてレビューできるようにするため。
- Pencil の MCP 接続(`mcp__pencil`)が有効であること。

## 手順

1. **要件を読む。** `docs/requirements/<機能名>.md` のスコープと受け入れ条件を確認する。
   デザインは受け入れ条件をすべて満たせる画面構成にする。
2. **Pencil でデザインを作成する。** `mcp__pencil` のツールを使い、対象画面のデザイン案を作る。
   - 画面に必要な要素（入力欄・ボタン・一覧・状態表示など）を受け入れ条件から洗い出す
   - 主要な状態を網羅する（例：タスクが0件のとき / 複数あるとき / エラー表示のとき）
3. **人間に見せてフィードバックをもらい、反復する。** 非エンジニアは見た目を見れば判断できる。
   口頭の「もっとこう」を Pencil 上で反映する。
4. **デザインの所在を要件ファイルに追記する。** どの画面のデザインがどこにあるか（Pencil 上の名前/URL等）を
   `docs/requirements/<機能名>.md` の末尾に「## デザイン」セクションとして記録する。

## 実装フェーズへの引き継ぎ（重要）

このスキルでは**デザインを作るだけ**です。コンポーネント化（コードにする）のは
scaffold 後の実装工程で行います。実装するときのルール：

- **shadcn/ui の素の部品（`components/ui/`）を土台にする。** ゼロから作らない。
- それらを組み合わせ、Pencil デザインに合わせた独自コンポーネントを `components/<feature>/` に作る。
- 色・余白・配置は Pencil デザインに合わせる。

## 完了の合図

- 受け入れ条件をすべて満たせる画面デザインができている
- 主要な状態（空・通常・エラー）がデザインに含まれている
- デザインの所在が要件ファイルに記録されている
- 人間がデザインに合意している
