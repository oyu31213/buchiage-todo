# テスト作成スキル

テストは**受け入れ条件を裏付けるため**に書きます。テストは2層で書き分けます。

| 層 | ツール | 置き場所 | 何をテストするか |
|---|---|---|---|
| E2E | Playwright | `tests/e2e/` | **UI の振る舞い**。受け入れ条件 1 つ = テスト 1 つ |
| unit | Vitest | `tests/unit/` | **バックエンドのロジックのみ**（Server Actions / Zod スキーマ / DB ヘルパー） |

> フロントのコンポーネント単体テストは**書きません**。UI の正しさは E2E が担保します。

## 最重要ルール：トレーサビリティ

各テストの冒頭コメントに、**そのテストがどの受け入れ条件・どのロジックに対応するか**を書きます。
これにより、コードを読めない人でも「どの条件が、どのテストで守られているか」を対応表として確認できます。

```ts
// 受け入れ条件: 「追加ボタンを押すと、入力したタスクが一覧の一番上に表示される」
// docs/requirements/todo.md の条件1
test("タスクを追加すると一覧の先頭に表示される", async ({ page }) => { ... })
```

## 手順

1. **要件を読む。** `docs/requirements/<機能名>.md` の受け入れ条件チェックリストを開く。
2. **E2E テストを書く（`tests/e2e/`）。** 受け入れ条件 1 つにつき 1 テスト。
   実際にブラウザを操作し、観測可能な振る舞いをアサートする。ログインが要る場合は Clerk のテスト用手段を使う。
3. **バックエンドの unit テストを書く（`tests/unit/`）。** Server Actions のロジック、
   Zod スキーマの検証（正常系・異常系）、DB ヘルパーの挙動を対象にする。外部依存はモック可。
4. **テストを実行して通す。**
   ```bash
   pnpm test:unit   # Vitest（プロジェクトのスクリプト名に合わせる）
   pnpm test:e2e    # Playwright（同上）
   ```
5. **対応表を確認する。** すべての受け入れ条件に、対応する E2E テストが1つ以上あることを確かめる。
   抜けがあれば追加する。

## 完了の合図

- すべての受け入れ条件に対応する E2E テストがある（コメントで対応が明示されている）
- バックエンドロジックの unit テストがある
- `pnpm test:unit` と `pnpm test:e2e` が両方通る

テストが通ったら、**`/code-review`** でコードを点検し、最後に**人間がブラウザで触って最終確認**します（MANUAL.md 参照）。
