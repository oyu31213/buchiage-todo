# 要件定義スキル

このスキルのゴールは、**非エンジニアでも完成を判断できる形**で要件を確定させることです。
実装方針は決めません。決めるのは次の2つだけです。

1. **スコープ** — 何を作るか / 作らないか
2. **受け入れ条件** — どうなれば「できた」と言えるか

## いちばん大事なルール：受け入れ条件は「観測可能な振る舞い」で書く

受け入れ条件は、コードを読めない人でもブラウザを触れば ○ / × を判定できる文にします。
この受け入れ条件は、後の **テスト** と **人間による最終確認** でそのまま使い回されます。

- ✅ 良い例:「追加ボタンを押すと、入力したタスクが一覧の一番上に表示される」
- ✅ 良い例:「未入力で追加ボタンを押すと『タスクを入力してください』と表示され、追加されない」
- ❌ 悪い例:「タスク管理がうまくできる」(何を見れば判定できるのか不明)
- ❌ 悪い例:「state を使って実装する」(これは実装方針。要件ではない)

## 手順

1. **目的を一言で確認する。** このアプリ / 機能は誰の何を解決するのか。
2. **曖昧な点は必ず質問する(勝手に仮定しない)。** 例:「タスクに期限は必要ですか？」「ログインは要りますか？」
   非エンジニアは細部を言語化していないことが多いので、ここで確認するほど後の手戻りが減ります。
3. **スコープ外を明示的に宣言させる。** 「今回はやらないこと」を書き出すのが最重要。
   例:「タグ付けは今回やらない」「他人とのタスク共有はやらない」。
4. **受け入れ条件をチェックリストにする。** 各条件は1つの振る舞い。チェックボックス形式で。
5. **`docs/requirements/<機能名>.md` に保存する。** 下のテンプレートを使う。

## 出力テンプレート（`docs/requirements/<機能名>.md`）

```markdown
# 要件: <機能名>

## 目的
<誰の何を解決するか。1〜2文>

## スコープ（やること）
- <作る機能を箇条書き>

## スコープ外（やらないこと）
- <今回やらないことを明示的に箇条書き>

## 受け入れ条件（= 完成の定義 / テストと最終確認はこれを使う）
- [ ] 条件1：<観測可能な振る舞い>
- [ ] 条件2：<観測可能な振る舞い>
- [ ] 条件3：<観測可能な振る舞い>

## 未確定・要確認
- <人間に確認したい点。なければ「なし」>
```

## 完了の合図

- スコープ外が1つ以上書かれている
- 受け入れ条件がすべて「観測可能な振る舞い」になっている
- 未確定の質問に人間が答え終わっている

これらが揃ったら、次は **design スキル** でデザインを作り、その後 Plan mode で作業計画を人間がレビューします。
（開発全体の流れは MANUAL.md を参照）
